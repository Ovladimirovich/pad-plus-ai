# Impulse Core v1 — план реализации

**Автор плана:** Grok (анализ репозитория + ревью предыдущего draft)  
**Дата:** 2026-07-17  
**Статус:** ✅ IMPLEMENTED (V1 runtime, 2026-07-17)  
**Цель:** сделать Impulse измеримым слоем runtime: state → pre-generate bias → generate inject → single update path → X-Ray → CI proof  
**Срок:** 5 рабочих дней (тонкий slice) + 1 optional day (research harness)

---

## 0. Зачем это делаем

Сейчас:

| Есть | Нет |
|------|-----|
| Core в `scripts/impulse.py` (4 измерения, JSON) | Core не в `backend/core/impulse/` |
| Event listener на `experience_captured` | Pre-generate read phase |
| `ImpulseUpdatePhase` (мёртвая: не в executor) | Inject в system prompt |
| Research в `experiments/I-00x` | X-Ray thought/decision |
| Unit-тесты core (248 строк) | CI, что bias реально уходит в LLM |

**Definition of Success (V1):**

1. На каждом non-trivial запросе в system prompt есть структурированный bias-блок импульса (если primary ≠ `unknown`).
2. X-Ray показывает primary + weights + факт инъекции.
3. Веса меняются **ровно одним** writer’ом после взаимодействия.
4. CI доказывает inject без live LLM.
5. README / ACTUAL_STATE больше не помечают Impulse как «частичный» по runtime.

**Не цель V1:** UI настроек, per-user impulse UI, decay, auto research A/B в CI, 6 провайдеров.

## 1. Принципы (не нарушать)

1. **Один writer deltas** — либо phase, либо listener; не оба.
2. **Schema v2 не ломать** — wire format `impulse.json` / `to_dict()` уже есть; не вводить второй `ImpulseState` с `dict[str,float]` как primary storage.
3. **Compat** — `scripts.impulse` остаётся thin re-export (тесты и CLI не падают).
4. **Hot path лёгкий** — ImpulsePhase = load/read, без LLM.
5. **CI без ключей** — inject proof на mock/unit; live eval — manual/nightly.
6. **Pattern как Emotion** — `PADModel` dual storage + `EmotionPhase` read-only.

## 2. Архитектура V1

```
                    ┌─────────────────────┐
                    │  ImpulseManager     │
                    │  PG (priority)      │
                    │  JSON (fallback)    │
                    └──────────┬──────────┘
                               │
         get_impulse_core()    │    apply_deltas()  [SINGLE WRITER]
                               ▼
┌──────────────┐   data    ┌──────────────┐   bias    ┌──────────────┐
│ ImpulsePhase │ ───────► │ ctx.context  │ ───────► │ GeneratePhase│
│ (pre-gen)    │           │ impulse_*    │           │ inject block │
└──────────────┘           └──────────────┘           └──────┬───────┘
                                                             │
                                                             ▼
                                                      response + X-Ray
                                                             │
                     ┌───────────────────────────────────────┘
                     ▼
            experience signal
            (interaction_type + significance)
                     │
                     ▼
            ┌────────────────────┐
            │ ImpulseUpdatePhase │  ← единственный writer
            │ apply + save       │
            └────────────────────┘

event_listener: НЕ применяет deltas (только optional metrics / deprecated stub)
```

### Почему phase, а не listener как writer

| | Phase | Listener |
|--|-------|----------|
| Видно в X-Ray request | да | нет / косвенно |
| Порядок относительно generate | явный | async race |
| Данные из ctx | можно гарантировать | зависит от bus |
| Двойной apply | если listener тоже пишет | риск |

**Решение V1:** `ImpulseUpdatePhase` = writer.  
`event_listener` → убрать apply deltas; оставить no-op log или удалить подписку до V2 (если bus понадобится для multi-process).

### Шаг 2. ImpulsePhase (pre-generate) (0.5d)

**Файл:** `backend/core/pipeline/phases/impulse.py`

```python
class ImpulsePhase(PipelinePhase):
    name = "impulse"

    async def execute(self, ctx) -> PhaseResult:
        try:
            core = get_impulse_core()
            state = core.to_dict()
            bias = core.get_bias_block()
            return PhaseResult(success=True, data={
                "impulse_state": state,
                "impulse_bias": bias,
                "impulse_primary": core.get_primary_label(),
                "impulse_prompt_line": core.get_prompt_line(),
                "impulse_active": [
                    {"label": d.label, "weight": d.weight, "question": d.question}
                    for d in core.get_active_questions(threshold=0.3)
                ],
            })
        except Exception as e:
            logger.warning(...); return PhaseResult(success=True, data={
                "impulse_state": {}, "impulse_bias": "", "impulse_primary": "unknown",
                "impulse_active": [],
            })
```

**Executor `_build_phases`:**

```python
("emotion", EmotionPhase()),
("impulse", ImpulsePhase()),      # NEW — после emotion, до persona
("persona", PersonaPhase()),
...
("generate", GeneratePhase()),
...
("emotion_update", EmotionUpdatePhase()),
("impulse_update", ImpulseUpdatePhase()),  # NEW registration
```

**Правила:**

| Правило | Значение |
|---------|----------|
| `_simple_skip` | **не** добавлять `impulse` (bias дешёвый и нужен) |
| `_independent_group` | **не** добавлять (последовательно, мгновенно) |
| `_stage_map` | `"impulse": "retrieve"` (рядом с emotion; **не** `"persona"`) |
| `impulse_update` stage | `"remember"` (как emotion_update) |

**Export** в `phases/__init__.py`.

### Шаг 5. Experience producer + ImpulseUpdatePhase (1d) — критический

Без этого update phase — мёртвый код.

#### 5.1 Producer (выбрать один, реализовать явно)

**Рекомендуемый MVP:** лёгкая эвристика **без LLM** в конце pre/post generate:

| Сигнал | `interaction_type` | `significance` |
|--------|--------------------|----------------|
| user feedback 👎 / contradiction markers | `criticism` / `contradiction` | 0.5–0.8 |
| user feedback 👍 / «спасибо» | `praise` | 0.4–0.7 |
| intent exploration / вопрос «почему/как» | `exploration` | 0.3–0.5 |
| pipeline error / fallback used | `error_recovery` | 0.5 |
| повтор того же вопроса (anti-loop near) | `repetition` | 0.3 |
| иначе | `new_knowledge` | 0.0–0.15 (skip if &lt; 0.2) |

**Где писать в ctx:**

- минимально: в `EvaluationPhase` или отдельный tiny helper `core/impulse/signals.py::infer_experience(ctx) -> tuple[type, sig]`
- вызывать **до** `ImpulseUpdatePhase` (evaluation уже есть перед save_episode; impulse_update поставить **после** evaluation)

Порядок post-generate (уточнённый):

```
generate → truth_loop → evaluation → save_episode → extraction
→ emotion_update → impulse_update → ...
```

Evaluation (или signals helper, вызванный из impulse_update в начале) **обязан** положить:

```python
ctx.context["experience_interaction_type"] = ...
ctx.context["experience_significance"] = ...
```

#### 5.2 ImpulseUpdatePhase

```python
# 1. если experience_* пусты — infer_experience(ctx)
# 2. apply_deltas(core, type, sig)
# 3. save manager
# 4. data: impulse_updated, impulse_state (after), deltas_applied
# 5. never fail request: success=True + log
```

#### 5.3 event_listener

```python
async def _on_experience_captured(...):
    # V1: DO NOT apply deltas (single writer = phase)
    logger.debug("impulse listener: ignored write (phase owns updates)")
```

Или полностью отписать до V2.  
Документировать в коде `WHY`.

### 15.3 Принципы (не нарушать)

1. **Никогда non-critical фаза не роняет request** — success=true + degradation info
2. **Один writer для каждого state** (паттерн из Impulse V1)
3. **PG = source of truth**, JSON/SQLite = fallback/cache
4. **Hot path минимален** — только фазы, влияющие на ответ
5. **Background = ControlTick или create_task** — не новый thread/middleware
6. **Измеримость прежде расширения** — eval harness до нового functional scope

### 15.4 P1 — Hot/background pipeline split

**Сейчас:** 24 фазы последовательно async/await. Фазы после generate:

```
evaluation → save_episode → extraction → emotion_update → impulse_update →
[consolidation] → persona_evolution → events → health → reflection →
dreams → metrics → response_guard
```

Ни одна не вызывает LLM. Совокупный overhead: 50–300ms.

**Цель:** response path = фазы, влияющие на ответ. Остальное — background.

**Изменения в executor.py:**

| Path | Фазы | Механизм |
|---|---|---|
| **Sync** | safety → intent → parallel group → impulse → persona → roots → identity → **generate** → truth_loop → evaluation → **response_guard** | as-is |
| **Deferred sync** | save_episode_light (id only) → extraction → emotion_update → impulse_update | await, быстрые |
| **Fire-and-forget** | consolidation → persona_evolution → reflection → dreams → health heavy | `asyncio.create_task` после ответа |
| **Background tick** | MetaLearner → ReflectionLoop → deep consolidation | ControlTick (300s, уже есть) |

**Детали:**

1. **save_episode** — разделить на `save_episode_light` (sync, возвращает episode_id) и `save_episode_deep` (async create_task)
2. **Фазы переезжают в fire-and-forget:**
   - `persona_evolution` (эвристики, <5ms)
   - `reflection` (чтение state, <5ms)
   - `dreams` (уже fire-and-forget, только счётчик)
   - `consolidation` (inline sync → create_task)
   - `health` (лёгкий assess → оставить; deep → tick)
3. **Остаются в sync path:** `extraction`, `emotion_update`, `impulse_update` — быстрые (<1ms), сохраняют state последовательно
4. **Response_guard** — остаётся последней sync-фазой (финальная проверка)
5. **correlation:** `session_id` + `trace_id` пробрасываются в фоновые задачи

**Оценка эффекта:** −30–50% wall time (2–4s → 1–2s на typical)

**Файлы для изменения:**
- `backend/core/pipeline/executor.py` — split pipeline, create_task для background
- `backend/core/pipeline/phases/save_episode.py` → `save_episode_light` + `save_episode_deep`
- `backend/core/tick/control_tick.py` — добавить deep consolidation/reflection
- `tests/test_pipeline/test_background.py` — новый

**Риски:**
- Background task упадёт → degradation незаметен (X-Ray + лог обязательны)
- Гонка save_episode_light → deep (id уже есть, deep дополняет)
- create_task не ждётся при shutdown → graceful wait
